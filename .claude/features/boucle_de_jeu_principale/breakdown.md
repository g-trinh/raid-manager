# Technical Breakdown ÔÇö `boucle_de_jeu_principale`

## 1. Overview

### Goals
- Implement the full playable game loop: Draft Ôćĺ Boss attempt Ôćĺ Inter-raid Ôćĺ Victory or Game Over
- 1 tier, 5 bosses, all core systems active (draft, boss resolution, rubber band, council, debrief, save)
- Online-required; snapshot saved after every council phase
- Static GDScript Resources for all game data (no content API)

### Non-Goals
- Conflict system (System 07) ÔÇö excluded from MVP
- Tier-end sequence (System 08) ÔÇö excluded from MVP
- Leaderboard UI in-game ÔÇö backend endpoint exists, no in-game screen
- Meta-progression / cross-run persistence
- Multiplayer or co-op

---

## 2. User Flows & Screens

### Flow: Full run
```
Main Menu Ôćĺ [POST /runs] Ôćĺ DraftScreen (8 draws)
  Ôćĺ TacticalChoiceScreen (pre-attempt choice)
    Ôćĺ BossResolutionScreen (outcome animation)
      Ôćĺ DebriefScreen (narrative)
        Ôćĺ CouncilScreen (3 events) Ôćĺ [PUT snapshot]
          Ôćĺ loop back to TacticalChoiceScreen (next boss or retry)
          OR Ôćĺ VictoryScreen (boss 5 killed)
          OR Ôćĺ GameOverScreen (attempt_count == 5)
```

### Screen states

| Screen | States |
|---|---|
| DraftScreen | draw_active / skip_disabled (0 skips left) / draft_complete |
| TacticalChoiceScreen | idle / choice_selected / confirming |
| BossResolutionScreen | resolving (brief animation) / outcome_kill / outcome_wipe |
| DebriefScreen | displaying / dismissed |
| CouncilScreen | idle (all 3 unresolved) / resolving (1 open) / chained (sub-choice open) / saving (snapshot in flight) / save_error (retry dialog) |
| VictoryScreen | displaying / dismissed |
| GameOverScreen | displaying / dismissed |
| HUD | always visible; updates on phase_changed |

### Error states
- **Network unavailable at run start**: show connection error overlay, block draft entry, retry button
- **Snapshot save failure**: block phase transition, show retry dialog in CouncilScreen
- **Snapshot load failure (resume)**: show error, offer "start new run" fallback

## 3. Functional Requirements

### Draft (Phase 0)
- 8 draws total; each draw presents exactly 3 `MemberData` profiles
- Player picks 1 per draw or skips (max 3 skips total across all draws)
- Skip counter is always visible; skip button disabled at 0 remaining
- Skipped draw advances without adding a member; the slot is filled by the next draw
- If all 8 slots are filled before 8 draws are exhausted, draft ends immediately
- Draft pool must guarantee at least 1 Tank, 1 Healer, and 3 DPS across all 24 drawn profiles (weighted generation)
- After draft_complete: `MemberRegistry.members` locked for the run; cannot be modified

### Tactical Choice (pre-attempt)
- Exactly 3 categories: Composition, Strategy, Consumable
- Player selects exactly 1 `TacticalChoiceData`; choice is irrevocable
- Consumable choices are one-use: decrement resource count; if resource is 0, choice is greyed out and unselectable
- Boss name, position (X/5), and attempt count (X/5) visible on this screen

### Boss Resolution
- Calls `BossResolver.resolve_attempt(members, boss, tactical_choice, rubber_band_active)`
- Resolution is instant; outcome displayed via brief animation (kill or wipe)
- On kill: distribute loot (+equipment points, weighted toward lowest-equipped member) and XP to all members
- On wipe: `attempt_count++`, `consecutive_wipes++`; check `RubberBandSystem` if `consecutive_wipes >= 3`
- On wipe with `attempt_count == 5`: emit signal to trigger GAME_OVER (no inter-raid)
- `rubber_band_active` flag is consumed and reset to `false` after resolution

### Debrief
- Text generated by `DebriefGenerator.generate(last_attempt_result, members)` using template + variable substitution
- Always shows: outcome (kill/wipe), underperformer name, reason category
- Player taps/clicks anywhere to dismiss

### Council (Inter-raid)
- `CouncilEventSystem.draw_events()` draws exactly 3 non-duplicate `EventCardData` from weighted pool
- Player sees all 3 simultaneously and chooses resolution order freely
- Each event: 2ÔÇô3 choices with visible immediate effect label
- Some events support chained choices (max 2 levels deep): first choice reveals second-level options
- Resolving event N can modify available choices on unresolved events (via effect application to RunData)
- After all 3 resolved: apply all effects to MemberRegistry/RunData, call NetworkManager.save_snapshot()
- Phase transition blocked until snapshot_saved signal received

### Win condition
- Boss 5 killed Ôćĺ after council Ôćĺ transition to VICTORY

### Game over condition
- `attempt_count == 5` on any boss Ôćĺ skip inter-raid Ôćĺ transition to GAME_OVER

### Save / Resume
- On app launch: `GET /runs/{id}` to check for active run; if found, offer resume
- Resume restores all Autoload state from snapshot JSON
- After victory/game-over: `POST /runs/{id}/complete` with outcome + final stats

### HUD
- Always shows: Boss X/5, Attempt X/5
- Updates immediately when GameState phase changes

## 4. Non-Functional Requirements

- **Performance**: Phase transitions must complete in < 300ms on target devices (web/mobile/desktop). Boss resolution is synchronous GDScript ÔÇö no async needed.
- **Responsiveness**: All UI via `Control` anchors + `Container` nodes. Baseline layout at 720p; scales to 4K without layout breaks.
- **Touch input**: All interactive elements minimum 44├Ś44px tap target on mobile.
- **Save reliability**: Snapshot save is blocking ÔÇö player cannot advance until backend confirms. Max 2 automatic retries before showing manual retry dialog.
- **Localization**: Not in scope for MVP. All strings hardcoded in French. String externalization (i18n) deferred.
- **Analytics**: Not specified. Recommended future addition: track boss kill/wipe per attempt, council choices made.
- **Accessibility**: Not specified. Recommended: all interactive elements focusable via keyboard; sufficient color contrast (WCAG AA).
- **Security**: Run state is client-authoritative. No server-side validation of scores in MVP. Note: this is exploitable ÔÇö flag for future anti-cheat pass.

## 5. Architecture & Data

See `architecture.md` for full layer decomposition, FSM, data model, and algorithm pseudocode. Summary below.

### State ownership
- All mutable run state lives in GDScript Autoloads (`GameState`, `RunData`, `MemberRegistry`)
- Backend stores opaque JSON snapshots; never interprets or validates game logic
- Scenes are stateless ÔÇö they read from Autoloads on `_ready()` and emit signals on user actions

### API endpoints (from `docs/api/openapi.yaml`)
| Method | Path | When called |
|---|---|---|
| `POST` | `/api/v1/runs` | Run start (after main menu) |
| `GET` | `/api/v1/runs/{id}` | App launch (resume check) |
| `PUT` | `/api/v1/runs/{id}/snapshot` | After every council |
| `POST` | `/api/v1/runs/{id}/complete` | Victory or game-over |
| `GET` | `/api/v1/leaderboard` | Post-run screen (optional for MVP) |

### Error handling
- All API calls in `NetworkManager.gd` wrapped with retry logic (2 retries, exponential backoff)
- `NetworkManager` emits `request_failed(endpoint, error)` signal; `PhaseController` handles by showing retry dialog
- Scenes never call HTTP directly

## 6. Component & UI Breakdown

### MemberProfileCard
- **Responsibilities**: Display member name, role icon, skill bar, morale bar, trait tags (2 positive, 1 negative)
- **Props**: `MemberData`; `selected: bool`; `mode: draft | roster`
- **Interactions**: tap to select (draft mode); tap to inspect (roster mode)
- **Reused in**: DraftScreen, TacticalChoiceScreen (composition tab)

### SkipCounter
- **Responsibilities**: Show remaining skips as visual dots (filled = available, empty = used)
- **Props**: `skips_remaining: int` (0ÔÇô3)
- **Interactions**: none (display only); parent DraftScreen disables skip button when 0

### CompositionPreview
- **Responsibilities**: Show 8 member slots (filled or empty) at bottom of DraftScreen
- **Props**: `members: Array[MemberData]` (partial during draft)
- **Interactions**: none in draft mode

### TacticalChoiceCard
- **Responsibilities**: Show choice label, category badge, effect hint; disabled state if resource depleted
- **Props**: `TacticalChoiceData`; `disabled: bool`; `selected: bool`
- **Interactions**: tap to select, tap again to deselect

### BossInfoPanel
- **Responsibilities**: Display boss name, position (X/5), current attempt (X/5), tier label
- **Props**: `BossData`; `attempt_count: int`
- **Reused in**: TacticalChoiceScreen, BossResolutionScreen

### OutcomeReveal
- **Responsibilities**: Animate kill (green flash, "Boss terrass├ę!") or wipe (red fade, "WIPE")
- **Props**: `outcome: String` ("kill" | "wipe"); `boss_name: String`
- **Interactions**: dismiss on tap/click after animation completes

### EventCard
- **Responsibilities**: Display category tag, title, description, 2ÔÇô3 choice buttons with visible effect labels
- **Props**: `EventCardData`; `resolved: bool`; `locked: bool` (while another card is being resolved)
- **Interactions**: tap choice to apply; if chained, reveals second-level choice panel inline
- **States**: unresolved / in-progress (first choice made, awaiting second) / resolved (greyed out)

### ChainedChoicePanel
- **Responsibilities**: Display second-level choices after a first choice is made on an EventCard
- **Props**: `choices: Array[Dictionary]`
- **Interactions**: tap to confirm; collapses EventCard into resolved state

### NarrativeText
- **Responsibilities**: Display debrief text with typewriter reveal animation
- **Props**: `text: String`
- **Interactions**: tap to skip typewriter animation; tap again to dismiss

### HUD
- **Responsibilities**: Always-on overlay showing Boss X/5 and Attempt X/5
- **Props**: derived from `GameState` + `RunData` via Autoload
- **Updates**: on `phase_changed` signal from `GameState`

### RetryDialog
- **Responsibilities**: Blocking overlay shown on snapshot save failure; "Retry" and (future) "Save locally" buttons
- **Props**: `error_message: String`
- **Interactions**: retry button re-calls `NetworkManager.save_snapshot()`

## 9. Risks, Edge Cases, and Mitigations

- **Boss difficulty miscalibration**: a boss with `difficulty_rating` set too high will reliably cause game-overs before the rubber band triggers. *Mitigation*: use `fixtures/member_median.tres` to validate that a median guild kills boss 1 Ôëą 70% of the time before shipping content data (TICK-014).

- **Draft pool exhaustion**: if the pool has fewer than 24 profiles, draws after the 8th would fail. *Mitigation*: TICK-014 requires exactly 24+ profiles; `DraftSystem` should assert pool size Ôëą 24 at init.

- **Snapshot race condition**: player closes the app during the save call. *Mitigation*: the snapshot sent to the backend includes the `phase` field set to the current phase. On resume, the client restores to that phase ÔÇö which for mid-council is `COUNCIL`, so the player re-draws 3 events. This is an acceptable UX tradeoff for MVP.

- **Council event pool too small**: with 30 events and 3 drawn per inter-raid over ~10 inter-raids per run, repetition is possible. *Mitigation*: track recently seen event IDs in `RunData`; exclude last 5 seen events from the draw pool. Low-cost implementation in `CouncilEventSystem`.

- **Consumable choices always exhausted early**: players spend potions on boss 1, leaving consumable tactical choices disabled for the rest of the run. *Mitigation*: resource replenishment events in the council pool (already in the 30-event set). Verify in playtesting.

- **Edge case ÔÇö skip to end of draft**: player uses 3 skips on draws 1ÔÇô3, then the pool may have fewer than 3 remaining unique profiles for later draws. *Mitigation*: pool size of 30+ ensures 24 unique profiles are available across all 8 draws even after skips burn through some.

- **Edge case ÔÇö game-over triggered on boss resolution, not council**: `attempt_count == 5` is checked in `BossResolutionScreen`, not `CouncilScreen`. PhaseController must handle this via the `attempt_resolved` signal before drawing council events. Verify in TICK-022 wiring.
